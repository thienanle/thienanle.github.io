Making it Easier for Users to Find Their Booked Course in a Community

The Khalibre Platform is a SaaS that was originally created to help large missions organizations manage and provide training for their workers. Its communities feature offers tools for collaboration, bringing people together around topics such as a team working on a project or a missions worker sharing his/her updates to supporters.

I joined the team when the basics of these features had already been built, and have since helped to improve existing areas, design new features, and establish a consistent and robust design language for the product.

Background

After I join a community, I can access member features such as posts and file sharing. On the Learning page, I can browse for courses that the community offers.

The My Learning page, which is global and sits outside of communities, allows me to see all the courses I’ve booked in one place, regardless of which community they are from.

When browsing a community’s courses, if I click on a course I’ve already booked, rather than being given the option to book the course, I get the option to “enter” the course, which takes me to the learning view.

The Problem

In a support ticket, a customer requested the following, paraphrased by me:

In a community, to find and enter a course after having previously booked it requires several extra steps.

  1. Users need to change the sort order from “most popular” to “alphabetical” each time and then find the course by name.
  2. After clicking on the booked course, user clicks on an activity but nothing happens. User needs to click on “enter course” and then can click on an activity.

Please change the default sort to “alphabetical” and skip the step of needing to click “enter course” again if the user has already booked the course.

Research

Question 1: why were users not using the My Learning page to find their booked courses?

This is what the My Learning page was designed for. Through conversations with the customer it quickly became evident that many users think of the platform in the context of communities. “I found and booked this course here, and I want to continue it here.” Even a seasoned user of the platform might find it more natural to access a course offered by a community by going to that community first.

Question 2: why the default sort was “most popular” in the first place, and is alphabetical sorting the answer?

I researched many websites and apps to understand why different sort options are used.

What I found:

  1. Alphabetical sorting is useful in cases where the names of items in a list are known. An example would be a list of people I’m connected with.

  2. On many sites, the user does not know the names of items in a list. The user is there to search or browse based on some criteria. The word the user is looking for might not be at the beginning of the title, so alphabetical sorting would not help, but would arbitrarily give prominence to titles that start with A or B or C. Therefore, the order should consider other outcomes, such as surfacing content that is most relevant to users.

“People rarely think A-Z” — Nielsen Norman Group

In our case, the community Learning page is a catalog of courses available for booking, where users don’t typically know the specific title of what they’re looking for because they’re browsing to discover content. If they did know the title, they would likely use search to find it.

The ideal sort would rely on an algorithm to surface content that is most relevant to users. The “most popular” sort became the default way to infer relevance in lieu of a more complex algorithm.

The Root Problem

The purpose of the Learning page in communities is to show the courses that are offered by the community. Understandably, learners are coming back here to find the courses they booked, and the page was not designed to facilitate this. Changing the sort order to alphabetical might make it easier to find my course, but only if I remember the name and the list of courses isn’t too long.

The Design

To solve the problem I found it necessary to 1) make it easier for users to easily find and identify courses that they already booked and 2) do this within the context of a community.

I wanted to create separation between booked and unbooked courses within the community, so that users don’t need to dig through one bucket to find the courses they’ve already booked. Here are two different design options:

The Result

The above shows what was finally implemented. It is like option 2, but rather than creating 2 separate lists, we introduced a new default sort option. “Most relevant” presents courses in an order that is more valuable to learners by considering booked courses as an additional criteria for relevance.

Notice also that the booked courses have a clear indication that they are booked. Clicking on these courses links to the learner’s course view rather than the booking page, so there is no need for an additional “enter course” step.

Learnings

Design is not an island. No matter how good a design is, collaboration must happen with development to ensure the project is feasible and realistic. Listening to others’ feedback is important.

Pick your battles! A potential issue can arise where if a user has booked many courses, unbooked courses lose the their discoverability. The final implementation doesn’t include a filter for booked and unbooked courses, which would help with the issue. Leaving this out would not have been my first choice, but it meant the team could deliver on time a solution that addresses the customer’s most immediate needs.

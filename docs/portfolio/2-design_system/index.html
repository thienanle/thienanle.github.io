Defining a Design System for an Existing SaaS Platform

The Khalibre Platform is a SaaS that was originally created to help large missions organizations manage and provide training for their workers. Its communities feature offers tools for collaboration, bringing people together around topics such as a team working on a project or a missions worker sharing his/her updates to supporters.

I joined the team when the basics of these features had already been built, and have since helped to improve existing areas, design new features, and establish a consistent and robust design language for the product.

The Problem

As with many digital products, the Khalibre Platform has seen its fair share of designers, developers, and product directors. When I joined the team, there were inconsistencies everywhere, from UI and UX to information architecture to code. I noticed redundancies in form elements and many definitions of nearly identical colors. These might not sound catastrophic, but in reality the issues across the system ran much deeper, and when added up they created a clumsy, confusing experience that did not instill confidence in users.

The screenshot above contains some examples of issues with:

  • Casing: some buttons and headings used all caps, while others used sentence or title case.
  • Information hierarchy: the title of the course is much smaller than the headings below it.
  • Icons: lack of harmony due to different stroke widths and sizes.

Goals

At the core of the issues was a lack of clearly defined standards that could be a single source of truth across teams. My goals in creating a design system were to

  • Increase consistency of the user experience.
  • Improve maintainability and scalability of components.
  • Speed up design and development by providing a library of flexible, predefined components and resources.
  • Streamline handoff and foster collaboration between teams.

The Process

I was not given dedicated time to work on the design system, but built it up over time alongside my day-to-day project work. Therefore, multiple and varying cycles of the process outlined below happened over time.

  1. Audit existing UI/UX elements and patterns in order to get a holistic view of redundancies and inconsistencies.
  2. Research other design systems in the industry to learn about what works.
  3. Consolidate and remove redundancies in existing components, variables, colors, and typography.
  4. Create new components, variables, colors, and typography to fill in gaps and make the design library more comprehensive.
  5. Collaborate with developers to achieve parity between design and code.
  6. Document standards and principles as guidelines for the team.

The Design System

Below are samples of the result of many iterations of the process I described in the previous section. The Khalibre Design System naturally continues to evolve with the product, but I would say it is at a place now where a majority of designs can be created using only existing components and patterns.

Stylistic Choices

When creating the design system, the Khalibre Platform was already a complex product with many screens and a large codebase. There were plans to rebuild some key areas, but the whole system would not be updated all at once.

The design system therefore adapted already-defined styles so that new pages could co-exist in harmony with what was already there. I also took the opportunity to refine and beautify many existing components.

The Component Library

Key principles that went into creating the components:

  • Components should only use sizes, colors, and typography predefined by the library for consistency and scalable future changes.
  • Provide enough variants and properties so that instance overrides and detaching are unnecessary 99% of the time.
  • Name properties so that components with equivalent properties retain their values if swapped.
  • Define fundamental interactions such as hover states so that they work out of the box.
  • Use instance swap properties only for nested instances that are designed to be swapped.
  • Expose properties from nested instances to minimize time drilling down.

Variables

Some design thinking that went into this:

  • Although a dark mode has yet to be implemented, variable names were chosen that would still make sense when a dark mode was added later.
  • Tokenization was used as an abstraction layer and to provide context for where variables should be used.
  • Typography considered consistent type scaling and specific line heights and paragraph spacing to achieve visual balance.

Parity With Code

The timing for me to begin working on the design system was good, partly because on the development side, the team was reaching a point where it was ready to move to a new framework for building the front end. I worked closely with (and sat next to) the lead developer on the project to produce components in Vue 3 that mirrored the definitions in Figma.

We also chose variable names that made sense both in Figma and in CSS. Using slashes in Figma makes it easy to organize and group variables, and also translates natively to the CSS dash naming convention.

Documentation

I created and continuously updated documentation that supplemented the definitions in Figma. This includes UI development documents that focus on the technical aspects related to the design system, copywriting documents that set standards for things like tone, casing, internationalization, and the use of AI, and a design team handbook that brings together all of the various design resources and helps to onboard future designers.
